**1. Main Meeting Content**

This meeting was held online through Zoom. The team mainly discussed the writing of **KC (knowledge concept)** content, the design of an initial user profile, the development of the Dashboard frontend and backend, this week’s task allocation, and GitHub branch management. The following decisions were made.

 

**2. Meeting Decisions**

**1. Confirming that KC content should be written by the team**

The team confirmed that the **KC (knowledge concept)** content in this project cannot be directly copied from websites, textbooks, or existing materials.

The KC content should be based on real learning scenarios. Team members need to understand the knowledge concepts first and then write them in their own words.

The requirements for KC writing include:

- The content should be based on real learning processes and     common knowledge points; 
- The content should not be directly copied from external     sources; 
- The content should show the team’s own understanding of the     knowledge concepts; 
- The explanation should be clear and easy to use for student     practice and ability analysis; 
- The KC content should match the project goal and support later     practice recommendation and user profiling. 

 

**2. Confirming the use of a diagnostic test during user login**

The team decided that users need to complete a **diagnostic test** during login or when they first use the system.

The main purposes of the diagnostic test are:

- To understand the user’s current knowledge level; 
- To identify which KCs the user has mastered well and which KCs     are weak; 
- To generate an initial learning profile for the user; 
- To help the system recommend more suitable practice content     later; 
- To help users find their weaknesses and practise more     effectively. 

The team agreed that the diagnostic test can be used as the starting point for personalised recommendation. This makes the project not only a normal practice platform, but also a platform that can analyse users’ learning situations and recommend exercises based on their needs.

 

**3. Confirming Dashboard development as this week’s main task**

The team confirmed that the main development task for this week is to complete the basic frontend and backend functions of the **Dashboard**.

The Dashboard is an important page where users can view their learning situation after entering the system, so the basic framework should be completed first.

The main Dashboard tasks for this week include:

- Building the frontend page framework of the Dashboard; 
- Implementing the basic page layout; 
- Designing areas for displaying the user’s learning situation; 
- Providing backend data interfaces required by the Dashboard; 
- Completing basic frontend-backend integration; 
- Ensuring that the Dashboard can display basic practice results     and KC mastery information. 

The team agreed that the Dashboard is an important module for displaying user profiles, practice results, and weak knowledge areas, so a basic version should be completed as early as possible.

 

**4. Confirming the completion of 8 KC contents this week**

The team decided to complete **8 KC contents** this week.

These KC contents will be used as the basic data for the early project demo. They will support the diagnostic test, practice recommendation, and Dashboard display.

The KC writing tasks include:

- Selecting 8 knowledge concepts that need to be completed first;     
- Clearly writing the definition, main understanding, and common     mistakes for each KC; 
- Making the content suitable for student learning and practice; 
- Keeping the structure of different KCs consistent; 
- Using the completed KC content for later question design and     user ability analysis. 

The team agreed that completing 8 KCs first can help the project quickly build a runnable demo and make it easier to expand more KCs later based on actual results.

 

**5. Confirming the use of different GitHub branches for development**

The team confirmed that different GitHub branches should be used for development to avoid confusion caused by uploading all files directly to the **main branch**.

The main purposes of branch management are:

- To prevent different members’ code from overwriting each other;     
- To reduce the risk of errors in the main branch; 
- To allow each member to complete their own tasks independently;     
- To make later code review and merging easier; 
- To keep the project structure clear and reduce collaboration     problems. 

The team decided that each member should develop on the corresponding branch according to their task. After the task is completed and checked, the code can then be merged into the main branch.
